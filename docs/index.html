Why QwenOS?

Four systems unified. 99.8% deduplicated. Clean OOP design.

Before — 4 Separate Systems

  • 18K+ files (95% duplicated node_modules)
  • 80+ overlapping commands
  • 3 separate AgentDB instances
  • 4 conflicting hook sets
  • 4 CLI entry points
  • MCP servers scattered everywhere

After — One Unified System

  • ~30 source files (clean, typed)
  • 25 deduplicated commands with aliases
  • 1 shared AgentDB + HNSW instance
  • 1 merged hook dispatcher
  • 1 qos CLI entry point
  • 1 MCP federation layer

Architecture

Three layers: CLI → Core → Infrastructure

⚡

CommandRouter

25 deduplicated commands with alias resolution. Async execution pipeline with pre/post hooks.

80→25 commands
🤖

AgentRegistry

105 agents across 15 categories. OOP lifecycle with spawn, execute, status, dispose.

105 agents
🧠

UnifiedMemory

3 memory systems merged. Namespaced keys, HNSW semantic search, access tracking.

3→1 instances
📡

EventBus

CRDT-based pub/sub for agent events. Local transport with distributed extension.

CRDT pub/sub
🔌

MCP Federation

Load-balanced access to external MCP servers. Health checking + automatic failover.

Multi-server
🎭

QwenOS Facade

Single entry point. initialize(), execute(), run(), status(), shutdown(). Hides all complexity.

OOP facade

Quick Start

30 seconds to running system

# Clone and install
$ git clone https://github.com/onthefox/QOS.git && cd QOS
$ bun install && bun run build

# Start using it
$ bun run start help
25 Commands (deduplicated from 80+)

$ qos run coder "create REST API"
Code implementation: create REST API

$ qos run security-reviewer "audit src/"
Security review: audit src/
